The Problem

OneSite is RealPage's flagship property management platform used by over 500,000 property managers across the US. It was built on .NET Framework 4.x with legacy System.Data.SqlClient, Windows authentication via NTLM, and a fragile hand-rolled deployment process.

As RealPage moved toward a cloud-first strategy, these technical debts became blockers: AAD authentication failures under IIS, SSL cert binding issues causing intermittent outages, and a deployment pipeline that required hours of manual steps — creating unacceptable risk for a platform of this scale.

"Fixing AAD Authentication & SSL Issues in IIS" ↗
🔐

AAD Auth Failures

Azure AD token validation was silently failing under IIS due to incorrect app pool identity and missing HTTPS redirect rules.

🔒

SSL Binding Issues

Multi-site IIS configs caused SSL cert binding conflicts, causing random HTTPS handshake failures in production.

🗄️

Deprecated SQL Driver

System.Data.SqlClient couldn't support AAD-integrated auth or modern TLS, blocking the cloud migration.

🚀

Fragile Deployments

Deploys required 4+ hours of manual steps, coordination across 3 teams, and a rollback plan that had never been tested.

🛠️
The Migration Strategy

The #1 constraint: zero unplanned downtime on a platform with 500K+ users. I designed a phased migration strategy using feature flags, blue-green deployment, and parallel-run verification at every step.

01

Phase 1 — SQL Driver Migration

Replaced all System.Data.SqlClient references with Microsoft.Data.SqlClient 4.x across 150+ files. Built a custom Roslyn analyzer to catch any future regressions. This unblocked AAD integrated auth and modern TLS 1.2+ support.

02

Phase 2 — IIS Hardening & SSL Fix

Audited all IIS site bindings and identified 3 conflicting SSL cert assignments. Implemented SNI-based SSL binding, enforced TLS 1.2, and disabled legacy cipher suites. Added HSTS headers and set up automated cert renewal via ACME protocol.

03

Phase 3 — Azure AD Integration

Migrated from NTLM to Azure AD using MSAL.NET with OIDC. Built custom middleware to handle token validation, tenant resolution, and claims transformation. Ran both auth systems in parallel for 4 weeks during cutover, using a feature flag to gradually shift traffic.

04

Phase 4 — CI/CD Pipeline Rebuild

Replaced manual deployment runbooks with Azure DevOps pipelines using blue-green deployment slots. Introduced automated smoke tests post-deploy and rollback triggers based on health check failures. Deploy time dropped from 4 hours to 25 minutes.

Technical Deep Dive

The most complex challenge was the AAD integration under IIS — specifically, why tokens validated perfectly in dev but failed in production.

Obtains AAD Bearer Token via MSAL
↓ HTTPS (TLS 1.2+)

IIS (SNI SSL Binding)
Cert resolved per hostname · HSTS enforced
↓ App Pool (ApplicationPoolIdentity)

JWT Validation → Tenant Resolution → RBAC → Controller
🗄️
Microsoft.Data.SqlClient
AAD Integrated Auth · Encrypted Connections · Modern TLS

The root cause of production failures: IIS app pool was running as NetworkService which couldn't access the certificate private key needed for AAD token signing validation. Switching to ApplicationPoolIdentity with explicit ACL grants resolved it instantly — a 2-week mystery solved in one afternoon.

  • Implemented token caching with distributed cache to reduce AAD round-trips by 80%
  • Added correlation IDs across all request logs for end-to-end traceability
  • Built a custom health check endpoint validating DB connectivity, AAD reachability, and cert expiry
  • Enforced claim-based authorization policies replacing legacy role tables in the database
